# プロジェクトの全体像と構造ガイド

## このアプリはどんな場面で使う？

- 社内で使う、**ダッシュボード型の業務アプリ** です。
- 主な用途は、**請求書・顧客・売上の状況をまとめて確認・管理すること** です。
- 想定ユーザー:
  - 経理担当: 「今月の売上や未払いの請求書を確認したい」
  - 営業担当: 「どの顧客にどれだけ請求が出ているか知りたい」
- 画面からできること（実装済み）:
  - ダッシュボードで **売上グラフ**（過去12ヶ月）や **最新の請求書一覧**（最新5件）を確認
  - ダッシュボードで統計カード（回収済み金額、未払い金額、請求書総数、顧客総数）を確認
  - 請求書ページで検索ボックスと請求書作成ボタンを表示

フロントから見ると「**社内向けの管理画面（管理ダッシュボード）**」だとイメージしておくと分かりやすいです。

## 技術スタック概要

このプロジェクトは、モダンなWebアプリケーション開発のための最新技術を採用しています。

### フルスタック技術（フロントエンド・バックエンド）

| 技術 | バージョン | 役割 |
|------|----------|------|
| **Next.js** | 15.4.2-canary.35 | **フロントエンドとバックエンドの両方を担当**。Reactベースのフルスタックフレームワーク。Server Componentsによりページコンポーネント内で直接データベース接続が可能。別のバックエンドサーバーは不要 |
| **React** | latest | UI構築のためのJavaScriptライブラリ |
| **TypeScript** | 5.7.3 | 型安全性を提供するJavaScriptの拡張 |
| **Tailwind CSS** | 3.4.17 | ユーティリティファーストのCSSフレームワーク |

### データベース技術

| 技術 | バージョン | 役割 |
|------|----------|------|
| **PostgreSQL** | - | リレーショナルデータベース（postgresパッケージ経由でNext.jsから直接接続） |

### その他の主要ライブラリ

- **@heroicons/react**: アイコンライブラリ
- **zod**: データバリデーション
- **clsx**: 条件付きCSSクラス名の管理
- **use-debounce**: デバウンス機能（検索などで使用）

## レイヤー構成

このプロジェクトは、役割ごとに **層（レイヤー）** に分かれています。各層は明確な責任を持ち、コードの整理と保守をしやすくしています。

```
┌─────────────────────────────────────────┐
│  プレゼンテーション層 (Presentation)    　　│
│  - UIコンポーネント (app/ui/)          　　│
│  - ページコンポーネント (app/*/page.tsx)　　│
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  ビジネスロジック層 (Business Logic)       │
│  - データ取得関数 (app/lib/data.ts)    　　│
│  - ユーティリティ関数 (app/lib/utils.ts)   │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  データアクセス層 (Data Access)     　　　　│
│  - PostgreSQL接続 (app/lib/data.ts）　　　│
│  - SQLクエリ実行 (app/lib/data.ts) 　　　　│
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  データベース層 (Database)        　　　　　│
│  - PostgreSQL                           │
└─────────────────────────────────────────┘
```

## 各レイヤーのざっくり説明

### 1. プレゼンテーション層（画面・UI）

- **場所**: `app/ui/`, `app/*/page.tsx`
- **やること**:
  - URLごとのページを定義（例: `app/dashboard/...` → `/dashboard`）
  - ボタン・テーブル・カードなどの見た目やレイアウトを作る
- **ポイント**:
  - Next.jsの**Server Components**で、画面コンポーネントの中から直接データ取得関数を呼べる
  - スタイルはほぼすべて**Tailwind CSS**のクラスで書く

### 2. ビジネスロジック層（アプリの「処理」）

- **場所**: `app/lib/data.ts`, `app/lib/utils.ts`
- **やること**:
  - DBからデータを取ってくる関数（`fetchRevenue`, `fetchLatestInvoices` など）
  - ページ表示用にデータを整える（合計を計算する、通貨フォーマットする など）
- **イメージ**:
  - 「どのデータを、どの条件で、どの形で画面に渡すか」を決める層
  - UIからは「関数を呼ぶだけ」で済むようにしている

### 3. データアクセス層（DBとのやりとり）

- **場所**: `app/lib/data.ts` の中
- **やること**:
  - `postgres` ライブラリでPostgreSQLに接続
  - SQLを書いてデータを取得・更新
- **ポイント**:
  - 接続情報は `process.env.POSTGRES_URL`（環境変数）から読む
  - `sql\`SELECT * FROM revenue\`` のようにテンプレートリテラルでSQLを書く

### 4. 型定義層（データの型定義）

- **場所**: `app/lib/definitions.ts`
- **やること**:
  - `User`, `Customer`, `Invoice`, `Revenue` など、DBや画面で使うデータの形をTypeScriptで定義
- **メリット**:
  - どんなデータが来るかがすぐ分かる
  - 間違った使い方をするとコンパイル時にエラーになって教えてくれる

### 5. ルーティング・レイアウト（ページの骨組み）

- **場所**:
  - ルーティング: `app/**/page.tsx`
  - 共通レイアウト: `app/layout.tsx`, `app/dashboard/layout.tsx`
- **やること**:
  - URLとページコンポーネントをひもづける
  - 全ページ共通のHTML構造やフォント、ダッシュボード用のサイドナビなどを定義

## データが動く流れ

1. ユーザーが `/dashboard` などのURLにアクセスする  
2. Next.jsが対応する `page.tsx` を実行し、そこで `fetchCardData()` などの関数を呼ぶ  
3. 関数の中でPostgreSQLにSQLを投げてデータを取得する  
4. 必要に応じてフォーマット・集計して、ページコンポーネントに渡す  
5. サーバー側でHTMLを生成してブラウザに返し、ユーザーに画面が表示される  

検索やページネーションも同じで、**URLパラメータ → `data.ts` の関数 → DB → 画面** という流れになっています。
